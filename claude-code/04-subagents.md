> [← 返回知识库首页](../README.md)

## Subagent 是什么？

**Subagent** 是一个拥有**独立上下文窗口**的专项 AI 助手，在隔离环境中完成任务后把结果汇报给主对话。

**类比**：你是项目经理，把一个大项目拆分成几个子任务，分别指派给不同的员工去做。每个员工有自己独立的工作台，互不干扰，完成后向你汇报结果。

---

## 为什么用 Subagent，而不是都写进 CLAUDE.md？

CLAUDE.md 可以给 Claude 背景知识，但它解决不了以下问题：

**上下文污染**：让 Claude 扫描整个代码库时，所有的探索过程和中间输出都会堆在主对话里，很快就把上下文窗口撑满。Subagent 在独立上下文中完成任务，只把结论汇报回来，主对话保持干净。

**无法并行**：CLAUDE.md 里的指令只能让 Claude 顺序执行。Subagent 可以多个同时跑，互不等待。

**无法限制工具**：在 CLAUDE.md 里写"只允许读取文件"，Claude 并不会真的被限制。Subagent 的 `tools` 字段是真实的权限控制，可以确保它不会误操作文件。

**模型选择**：不同任务对模型能力要求不同。CLAUDE.md 无法为不同任务指定不同模型，Subagent 的 `model` 字段可以精细控制。

| | CLAUDE.md | Subagent |
|--|-----------|----------|
| 上下文隔离 | 无，所有输出都在主对话 | 独立上下文，只返回结论 |
| 并行执行 | 不支持 | 支持多个同时运行 |
| 工具限制 | 仅建议，不强制 | `tools` 字段真实限权 |
| 模型控制 | 无 | `model` 字段按任务指定 |
| 适合场景 | 静态背景知识，每次都需要 | 独立任务，需要隔离或并行 |

## 如何触发 Subagent

**方式一：Claude 自动识别并调用**

Claude 根据你的请求内容，判断是否需要派遣 Subagent。比如让它分析一个大型代码库，它会自动调用 Explore Subagent 去扫描。

**方式二：用户用 @ 语法直接点名**

```
@"code-reviewer" 帮我看看这段代码有没有安全漏洞
```

**方式三：自然语言描述**

```
用 Explore agent 扫一下 ./src 目录的结构
```

## 内置 Subagent 类型

| Subagent | 工具权限 | 适合做什么 |
|----------|---------|-----------|
| `general-purpose` | 全部工具 | 搜索代码、多步骤执行、复杂综合任务 |
| `Explore` | 只读（无 Write/Edit）| 快速探索代码库，不会修改任何文件 |
| `Plan` | 只读 | 调研背景信息、制定实施方案 |
| `test-agent` | 全部工具 | 写测试、调试失败用例、提升覆盖率 |

## 如何自定义一个 Subagent

自定义 Subagent 是一个 `.md` 文件，放在 `agents/` 目录下，格式与 Skill 类似——顶部 frontmatter 配置，下面写系统提示词。

**存放位置：**

| 路径 | 作用范围 |
|------|---------|
| `~/.claude/agents/<name>.md` | 个人全局，所有项目可用 |
| `.claude/agents/<name>.md` | 项目级，可提交到 Git 共享给团队 |

**示例：自定义一个"竞品分析 Subagent"**

文件路径：`~/.claude/agents/competitive-analyst.md`

```markdown
---
name: competitive-analyst
description: >
  竞品分析专家。当用户需要分析竞品功能、对比产品差异、
  或调研市场上的同类解决方案时调用。
tools: WebSearch, WebFetch, Write
model: sonnet
---

你是一位经验丰富的产品竞品分析师。

收到分析请求后：
1. 先确认分析维度（功能、定价、用户体验、市场定位）
2. 通过网络搜索收集竞品最新信息
3. 用结构化表格对比各维度
4. 最后给出 2-3 条对我方产品的启示

输出格式：
- 用 Markdown 表格对比
- 结论部分用"我们可以借鉴的点"和"我们的差异化优势"两个小节
- 语言简洁，避免堆砌描述
```

**常用 frontmatter 字段：**

| 字段 | 说明 |
|------|------|
| `name` | Subagent 名称，用于 @ 调用 |
| `description` | 触发场景描述，Claude 靠这个判断何时自动调用 |
| `tools` | 允许使用的工具，不填则继承全部工具 |
| `model` | 使用的模型（`haiku` 省钱、`sonnet` 均衡、`opus` 最强）|
| `skills` | 要注入的 Skill 名称列表，启动时将 Skill **全文**注入上下文（见下方说明）|
| `permissionMode` | 权限模式（`default` / `acceptEdits` / `bypassPermissions`）|
| `isolation` | 设为 `worktree` 则在隔离的 git 工作区运行 |

## Skill vs Subagent 对比

### 加载机制

```
主 Agent 启动时：
┌─────────────────────────────────────────────────────┐
│  加载进上下文（每次都有）                              │
│  ├── CLAUDE.md 全文                                  │
│  ├── 所有 Skill 的 description 字段（仅几十个字）     │
│  └── 所有 Subagent 的 name + description（仅元数据）  │
└─────────────────────────────────────────────────────┘
          │                              │
          │ 关键词触发 Skill              │ 派遣 Subagent
          ▼                              ▼
┌─────────────────────┐    ┌──────────────────────────────────┐
│ SKILL.md 全文        │    │ 新建独立上下文（不继承主对话）       │
│ + 脚本/模板          │    │ ├── 注入 agent .md 全文           │
│ 注入当前主对话        │    │ ├── 若指定 skills：               │
└─────────────────────┘    │ │   注入各 Skill 全文（非描述）    │
                           │ └── 在子进程运行，结论汇报回主对话  │
                           └──────────────────────────────────┘
```

Subagent 没有"按关键词自动触发 Skill"的机制，无法继承主对话已加载的 Skill。如果需要让 Subagent 拥有某个 Skill 的知识，必须在 frontmatter 里用 `skills` 字段**显式列出**，Claude 会在启动时将其全文注入。

这与主 Agent 的处理方式截然不同：

| | 主 Agent | Subagent（指定 `skills`）|
|--|---------|------------------------|
| 启动时加载 | 只加载 description（几十字）| 直接注入 Skill **全文** |
| 全文何时加载 | 关键词触发后 | 启动时已全量注入 |
| Skill 数量多时 | 不影响启动，按需加载 | 全量注入，占用上下文，按需添加 |

**`skills` 字段用法示例：**

```yaml
---
name: api-developer
description: 按团队规范实现 API 接口
skills:
  - api-conventions
  - error-handling-patterns
---

实现 API 接口时，遵循已注入的规范和错误处理模式。
```

**为什么新增 Skill 后需要重启 Claude？**

主 Agent 的 Skill 描述在 **Claude 启动时一次性扫描加载**，运行过程中不会重新扫描。新建 Skill 文件夹后，必须重启 Claude 才能被感知和触发。

逻辑链条：**启动时扫描 → 运行中不再扫描 → 新增文件无感知 → 需重启**。

| 担心的问题 | 实际情况 |
|-----------|---------|
| 装了 20 个 Skill，启动会很慢？ | 不会，只加载描述，几十 token |
| Subagent 做了大量分析会塞满对话？ | 不会，独立运行，只返回结论 |
| CLAUDE.md 写多了有影响？ | 有，每次全量加载，建议控制在 200 行内 |
| Skill 的 SKILL.md 写很长？ | 触发时才加载，不触发没影响 |

### 核心差异

| 维度 | Skill | Subagent |
|------|-------|---------|
| **上下文** | 在主对话内展开，共享上下文 | 独立上下文，不占用主窗口 |
| **调用方式** | 关键词自动触发 / `/skill-name` | Claude 自动调用 / 用户 `@"name"` |
| **文件结构** | 文件夹（`SKILL.md` + 资源文件）| 单个 `.md` 文件 |
| **适合输出体量** | 轻量，需要和用户来回交互 | 大量输出、详细分析，最后汇总结果 |
| **工具限制** | `allowed-tools` 字段控制 | `tools` 字段控制，可精细限权 |
| **并行执行** | 不支持 | 支持，多个 Subagent 可同时工作 |

### 怎么判断该用哪个？

**选 Skill，当你需要：**
- 把某段知识或规范**持续注入**到对话（如 PRD 写作规范、API 命名约定）
- 任务需要**频繁和用户来回确认**（Skill 运行在主对话，交互更顺畅）
- 触发一个**固定流程模板**（如 `/commit` 生成提交信息）

**选 Subagent，当你需要：**
- 任务会产生**大量输出**，不想塞满主对话（如扫描整个代码库）
- 需要**并行处理**多个独立子任务（如同时分析 3 个竞品）
- 需要**限制工具权限**（如只读的代码审查，不让它改任何文件）
- 任务是**自包含的**，最后只需要一个结论汇报回来

**口诀：**
> 需要边做边聊 → **Skill**；丢出去跑完汇报 → **Subagent**
